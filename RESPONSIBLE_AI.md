# Responsible AI in This Sample

This sample demonstrates AI-assisted clinical workflows using Amazon Connect
Health, Amazon Bedrock, and AWS HealthLake. AI-generated clinical content
carries real risk to patient safety if used without appropriate guardrails
and human oversight. This document explains the Responsible AI commitments
embedded in this sample and the corresponding obligations for any production
deployment built on this code.

## Core Principles

### 1. Human-in-the-Loop is Mandatory

Every AI-generated clinical artifact in this sample, SOAP notes, ICD-10 and
CPT codes, patient insights summaries, after-visit summaries, and Bedrock
synthesized narratives, is shown to a licensed healthcare provider for
review and explicit approval before being persisted to the patient record or
delivered to the patient.

In the user interface, this is enforced by:

- An `Edit Notes` button on the SOAP overlay (`frontend/index.html`) that
  allows the clinician to modify any AI-generated content
- An `Approve & Save` button that is the only path to persist clinical data
  to AWS HealthLake (creating Encounter, DocumentReference, and Condition
  resources)
- A prominent AI-generated-content disclaimer banner on every screen
  displaying AI output

The disclaimer text reads:

> ⚠️ **AI-Generated Content:** This documentation was generated by AI and
> requires physician review and approval. Do not use as the sole basis for
> clinical decisions.

This disclaimer is required and must remain visible in any derived
deployment.

### 2. Amazon Bedrock Guardrails Are Required for Production

The backend services in this sample invoke Amazon Bedrock to synthesize
patient narratives, visit priorities, and pre-visit summaries. Each
invocation conditionally attaches a Bedrock Guardrail when the
`BEDROCK_GUARDRAIL_ID` environment variable is set:

```python
response = bedrock.invoke_model(
    modelId=BEDROCK_MODEL_ID,
    contentType='application/json',
    accept='application/json',
    body=body,
    **({
        "guardrailIdentifier": BEDROCK_GUARDRAIL_ID,
        "guardrailVersion": BEDROCK_GUARDRAIL_VERSION,
    } if BEDROCK_GUARDRAIL_ID else {})
)
```

A ready-to-deploy healthcare guardrail template is provided at:

```
shared/bedrock-guardrails/healthcare-guardrail.yaml
```

The template configures:

- **Content filters**: HIGH strength on SEXUAL, VIOLENCE, HATE, INSULTS,
  MISCONDUCT, and PROMPT_ATTACK categories
- **PII redaction**: ANONYMIZE on NAME, EMAIL, PHONE, ADDRESS, AGE, USERNAME,
  IP_ADDRESS, MAC_ADDRESS; BLOCK on SSN, BANK_ACCOUNT, CREDIT_CARD,
  DRIVER_ID, PASSWORD, US_PASSPORT
- **Denied topics**: unsupervised medical diagnosis, unsupervised treatment
  recommendation, and emergency-response misdirection
- **Word filters**: blocks phrases that bypass human-in-the-loop requirements

Deploy this template, take the `GuardrailId` and `GuardrailVersion` outputs,
and set them as environment variables on the backend ECS task.

**Operating the sample without Guardrails in any environment that processes
real patient data is not supported and is explicitly out of scope of this
sample's responsible-AI commitments.**

### 3. Output Validation Is Required Beyond This Sample

The sample shows how to display AI-generated medical codes (ICD-10, CPT) and
SOAP notes for clinician review. It does not validate codes against current
coding standards or enforce business rules (e.g., "this CPT code requires
that ICD code as a justification"). A production deployment must add:

- ICD-10 and CPT code validation against the current code set
- Cross-referencing of codes against the documented encounter
- Reasonableness checks on values (e.g., flagging a medication dose outside
  expected ranges)
- Logging of clinician modifications (what was changed from the AI output
  before approval)

### 4. Bias Monitoring Is the Operator's Responsibility

AI clinical models can carry bias across patient demographics. This sample
does not include automated bias evaluation. Production deployments should:

- Periodically sample AI outputs across patient demographics and audit for
  systematic differences in care recommendations
- Monitor approval / modification rates by patient demographic to detect
  cases where AI outputs are disproportionately edited for certain groups
- Engage clinical leadership in periodic review of AI behavior on edge cases

### 5. Audit Logging Is Required

The sample logs metadata (event counts, resource IDs, session IDs) but does
not log PHI. Production audit requirements typically demand:

- Logging of every AI invocation with the model ID, guardrail ID, and a
  reference to the request (without logging the prompt or response content)
- Logging of every clinician approval, with timestamp, clinician identity,
  and a hash of the approved content
- Long-term immutable storage of these audit logs (e.g., S3 with Object Lock)
- Periodic review by compliance teams

## What This Sample Does Not Do

To set expectations clearly, this sample explicitly does not:

- Implement Amazon Bedrock Guardrails by default (the guardrail ID is blank
  unless you set it; this enables `python3 server.py` demo mode without
  AWS credentials)
- Validate ICD-10 or CPT codes against current code sets
- Implement automated bias detection
- Implement long-term audit logging (relies on CloudWatch defaults)
- Encrypt PHI at the application layer (relies on AWS service defaults —
  HealthLake at rest, TLS in transit)
- Implement role-based access control beyond IAM (e.g., role-based UI
  features for different clinician types)
- Implement provider attestation workflows or electronic signature standards
  (e.g., 21 CFR Part 11)

Each of these is appropriate for a production deployment and out of scope
for a published sample.

## Required Disclaimers in the UI

The following disclaimers are required and present in the sample. They must
remain visible in any production deployment:

| Location | Disclaimer |
|---|---|
| SOAP notes overlay header | "⚠️ AI-Generated Content: This documentation was generated by AI and requires physician review and approval. Do not use as the sole basis for clinical decisions." |
| Pre-visit insights iframe | "⚠️ AI-generated content for clinical reference only. All information must be verified by healthcare professionals before use in patient care." |
| Backend API responses | All Bedrock-invoking endpoints return a `disclaimer` field: "AI-generated clinical content requires review by a licensed healthcare professional before use in patient care." |

## Compliance Notes

This sample is provided as a starting point. A production deployment must
satisfy, at minimum:

- **HIPAA** Privacy and Security Rules, including a Business Associate
  Addendum (BAA) with AWS covering all services used
- **HITECH** breach notification requirements
- **FDA** regulations governing clinical decision support, where applicable
  (this sample provides documentation support, not diagnostic decision
  support, but local interpretation may vary)
- State medical-licensing requirements for the jurisdictions where care is
  delivered
- For US Medicare/Medicaid: applicable CMS regulations on AI-assisted
  documentation and coding

This list is illustrative, not exhaustive. Consult your compliance and legal
teams before any production deployment.
